## **Теоретичні відомості**

### **Платформа Node-RED**

**Node-RED** — візуальне середовище програмування на базі Node.js, призначене для побудови потоків обробки даних шляхом з'єднання готових блоків — нод. Платформа особливо поширена в задачах інтеграції сервісів, обробки телеметрії та побудови систем моніторингу.

Програма у Node-RED — це **потік (flow)**: сукупність нод, з'єднаних провідниками (wires). Дані рухаються від ноди до ноди у вигляді JavaScript-об'єктів, що називаються **повідомленнями (`msg`)**.

### **Об'єкт повідомлення `msg`**

Об'єкт `msg` — це звичайний JavaScript-об'єкт, що передається між нодами. Кожна нода отримує `msg` на вхід, може його модифікувати, і передати далі. Стандартні поля:

| Поле          | Призначення                                        |
| ------------- | -------------------------------------------------- |
| `msg.payload` | Основний вміст повідомлення (число, рядок, об'єкт) |
| `msg.topic`   | Тема — для маршрутизації або групування            |
| `msg._msgid`  | Унікальний ідентифікатор (генерується автоматично) |

Розробник може додавати власні поля: `msg.sensor`, `msg.timestamp`, `msg.user` тощо.

### **Ключові ноди ядра**

| Нода       | Призначення                                           |
| ---------- | ----------------------------------------------------- |
| `Inject`   | Ініціює потік: вручну, періодично або за розкладом    |
| `Debug`    | Виводить вміст `msg` на бічну панель для перегляду    |
| `Function` | Виконує довільний JavaScript-код для обробки `msg`    |
| `Change`   | Встановлює, переміщає або видаляє поля `msg` без коду |
| `Switch`   | Розгалужує потік на основі значення поля `msg`        |
| `Template` | Формує рядок за шаблоном Mustache зі значень `msg`    |

### **Робочий цикл у Node-RED**

1. Відкрити редактор у браузері (зазвичай `http://localhost:1880`)
2. Перетягнути ноди з палітри на полотно (workspace)
3. Налаштувати ноди (подвійним кліком відкрити параметри)
4. З'єднати ноди провідниками (від вихідного порту однієї до вхідного іншої)
5. Натиснути **Deploy** для розгортання змін
6. Запустити потік (натиснути кнопку зліва від Inject)
7. Перевірити результат на бічній панелі Debug

> **Ключовий момент:** будь-які зміни у редакторі не діють, доки не натиснуто червону кнопку **Deploy** у правому верхньому куті.

### **Експорт потоків у JSON**

Будь-який потік можна експортувати як JSON-текст для збереження або передачі:

1. Виділити ноди на полотні (рамкою або `Ctrl+A`)
2. Меню `≡` → **Export**
3. Скопіювати JSON у буфер обміну або зберегти у файл

Імпорт виконується аналогічно через меню `≡` → **Import**.

#### **Завдання 2: Function-нода — обробка значення (flow-2.json)**

Створіть потік, що періодично генерує випадкове значення «тиску» у трубопроводі, перетворює його у структурований об'єкт та виводить на Debug. Function-нода також повинна відображати поточне значення у вигляді статусу під нодою.

**Структура потоку:**

```
[Inject "Кожні 3 сек"] ──▶ [Function "Генератор тиску"] ──▶ [Debug "Показники"]
```

**Налаштування Inject:**

- `Name`: `Кожні 3 сек`
- `msg.payload`: тип `string`, значення `start`
- `Repeat`: `interval`, кожні **3 seconds**
- `Inject once after`: `0.1` seconds (щоб запуститись одразу після Deploy)

**Код Function-ноди «Генератор тиску»:**

```javascript
// Генеруємо випадкове значення тиску від 0.5 до 8.5 бар
const pressure = +(0.5 + Math.random() * 8).toFixed(2);

// Формуємо структурований payload
msg.payload = {
  device: "P-204",
  pressure: pressure,
  unit: "bar",
  timestamp: new Date().toISOString(),
};

// Перевизначаємо topic для маршрутизації
msg.topic = "pipeline/pressure";

// Відобразити поточне значення у статусі під нодою
node.status({
  fill: "blue",
  shape: "dot",
  text: "Тиск: " + pressure + " bar",
});

return msg;
```

**Налаштування Debug:**

- `Output`: `msg.payload` (за замовчуванням)

**Що потрібно перевірити після Deploy:**

1. Кожні 3 секунди на панелі Debug з'являється новий об'єкт `payload` з різними значеннями `pressure`.
2. Під Function-нодою на полотні відображається синій кружечок зі значенням останнього тиску.
3. Поле `msg.payload` містить об'єкт з ключами `device`, `pressure`, `unit`, `timestamp`.

**Експерименти:**

- Натисніть зелену кнопку зліва від Debug-ноди, щоб тимчасово вимкнути її. Перевірте, що повідомлення перестали з'являтися. Увімкніть знову.
- Натисніть Inject вручну — потік запуститься позачергово.

**Експорт:** збережіть потік у `flow-2.json`.

**Контрольні точки виконання:**

- Function-нода щоразу видає число тиску в межах 0.5–8.5
- Статус під Function-нодою оновлюється на кожному запуску
- На Debug з'являється структурований об'єкт, а не просто число

#### **Завдання 3: Change + Switch + Template — класифікація рівня заряду (flow-3.json)**

Створіть потік, що приймає рівень заряду батареї пристрою, класифікує його (низький / середній / повний) та формує текстове повідомлення з рекомендацією. Завдання демонструє роботу з трьома нодами одночасно: Change, Switch, Template.

**Структура потоку:**

```
[Inject "0%"]    ┐
[Inject "45%"]   ├──▶ [Change "Підготовка"] ──▶ [Switch "Класифікація"]
[Inject "78%"]   │                                      │
[Inject "100%"]  ┘                                      ├──▶ [Template "Низький"]  ──┐
                                                        ├──▶ [Template "Середній"] ──┼──▶ [Debug]
                                                        └──▶ [Template "Повний"]   ──┘
```

**Налаштування 4-х нод `Inject`:**

| Ім'я   | `msg.payload` (тип `number`) |
| ------ | ---------------------------- |
| `0%`   | `0`                          |
| `45%`  | `45`                         |
| `78%`  | `78`                         |
| `100%` | `100`                        |

> Усі чотири ноди Inject з'єднайте з одним входом ноди Change.

**Налаштування ноди Change «Підготовка»:**

Додайте дві операції (кнопка `+ add`):

1. `Set` `msg.battery` `to` (обрати тип `msg.`) `payload`
2. `Set` `msg.topic` `to` (обрати тип `string`) `device/battery`

> У першій операції тип значення — `msg.`, що означає «взяти значення з іншого поля `msg`». У другій операції тип — `string`, тобто пряме рядкове значення. Після Change у `msg` з'явиться поле `battery` з копією значення `payload`, а саме `payload` залишиться числом.

**Налаштування ноди Switch «Класифікація»:**

- `Property`: `msg.battery`
- Режим: `stopping after first match`
- Правила (порядок важливий):

| № правила | Умова                            | Куди йде msg       |
| --------- | -------------------------------- | ------------------ |
| 1         | `<` `20`                         | Вихід 1 (Низький)  |
| 2         | `is between` `20` `80` (включно) | Вихід 2 (Середній) |
| 3         | `>` `80`                         | Вихід 3 (Повний)   |

> **Важливо:** Switch створює вихідні порти у тому ж порядку, що й правила. Перше правило → верхній (перший) вихід, друге → середній, третє → нижній. При з'єднанні з Template-нодами уважно з'єднуйте кожен вихід із відповідною Template-нодою.

**Налаштування 3-х нод Template:**

| Ім'я       | З'єднати з виходом Switch | Шаблон (Property: `msg.payload`)                               |
| ---------- | ------------------------- | -------------------------------------------------------------- |
| `Низький`  | Вихід 1                   | `⚠️ Заряд {{battery}}% — потрібна термінова зарядка пристрою`  |
| `Середній` | Вихід 2                   | `🔋 Заряд {{battery}}% — пристрій працює в нормальному режимі` |
| `Повний`   | Вихід 3                   | `✅ Заряд {{battery}}% — пристрій повністю заряджений`         |

**Налаштування Debug:**

- `Output`: `msg.payload`

**Очікувані результати на панелі Debug** (після почергового кліку по 4-х Inject):

```
⚠️ Заряд 0% — потрібна термінова зарядка пристрою
🔋 Заряд 45% — пристрій працює в нормальному режимі
🔋 Заряд 78% — пристрій працює в нормальному режимі
✅ Заряд 100% — пристрій повністю заряджений
```

**Експорт:** збережіть потік у `flow-3.json`.

**Контрольні точки виконання:**

- Кожне натискання Inject видає одне повідомлення відповідного типу
- Значення `45` та `78` обидва потрапляють у гілку «Середній»
- Значення `0` потрапляє в «Низький», а `100` — у «Повний»
- На жодному кроці не використовувалась Function-нода

#### **Завдання 4: Комплексний потік за варіантом (flow-4.json)**

Створіть потік, що поєднує **усі вивчені ноди** (Inject, Function, Change, Switch, Template, Debug), для предметної області з вашого варіанту (визначається за номером у журналі).

| Номер варіанта | Предметна область                                 |
| -------------- | ------------------------------------------------- |
| 1, 11, 21      | Швидкість обертання вентилятора серверної (RPM)   |
| 2, 12, 22      | Вологість повітря у складському приміщенні (%)    |
| 3, 13, 23      | Рівень води у резервуарі (%)                      |
| 4, 14, 24      | Потужність споживання електроенергії (кВт)        |
| 5, 15, 25      | Концентрація CO₂ у приміщенні (ppm)               |
| 6, 16, 26      | Освітленість на робочому місці (люкс)             |
| 7, 17, 27      | Рівень шуму у виробничому цеху (дБ)               |
| 8, 18, 28      | Швидкість потоку рідини в трубопроводі (л/хв)     |
| 9, 19, 29      | Заповнення дискового простору сервера (%)         |
| 10, 20, 30     | Кількість активних з'єднань мережевого обладнання |

**Обов'язкові вимоги до потоку:**

1. **Inject** — періодичний (кожні 5 секунд), генерує початкове значення (можна довільне, головне щоб був тригер).
2. **Function** — генерує об'єкт `payload` з полями `device`, `value`, `unit`, `timestamp`. Значення `value` має бути випадковим у діапазоні, відповідному предметній області (наприклад, для RPM — від 1000 до 5000). Function має також встановлювати `node.status` із поточним значенням.
3. **Change** — переносить значення у спеціальне поле для подальшої перевірки (наприклад, `Set msg.measurement to msg.payload.value`).
4. **Switch** — розгалужує потік мінімум на **3 виходи** на основі класифікації значення (норма / попередження / критичне). Правила обираєте самостійно з урахуванням предметної області.
5. **Template** — для кожної гілки після Switch створіть окрему Template-ноду, що формує повідомлення з emoji, відповідним для рівня (наприклад, ✅ / ⚠️ / 🚨), та значеннями з `msg` через синтаксис Mustache.
6. **Debug** — спільний для всіх гілок, виводить кінцевий рядок `msg.payload`.

**Приклад порогів для варіанту «концентрація CO₂»:**

- Норма: до 800 ppm — `✅ Якість повітря в нормі`
- Попередження: 800–1400 ppm — `⚠️ Якість повітря погіршується, потрібне провітрювання`
- Критично: понад 1400 ppm — `🚨 Критичний рівень CO₂, необхідно негайно провітрити приміщення`

**Експорт:** збережіть потік у `flow-4.json`.

**Контрольні точки виконання:**

- Потік запускається автоматично кожні 5 секунд
- За тривалої роботи (1–2 хвилини) на Debug з'являються повідомлення з усіх трьох гілок Switch
- Function відображає поточне значення у статусі під нодою
- Жодна нода не залишилась без імені (`Name`)

### **Етап 3: Перевірка**

## **Типові помилки та їх вирішення**

| Проблема                                                   | Рішення                                                                                                                                                                |
| ---------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Зміни у потоці «не діють», потік не оновлюється            | Натисніть **Deploy** (червона кнопка). Будь-які зміни не діють до Deploy                                                                                               |
| Function-нода не передає повідомлення далі                 | Перевірте, що код завершується рядком `return msg;`                                                                                                                    |
| Debug-панель порожня, нічого не з'являється                | Розгорніть бічну панель (іконка 🐞 праворуч). Переконайтесь, що Debug-нода активна (зелений значок)                                                                    |
| Mustache-вставка виводить порожній рядок                   | Перевірте шлях у фігурних дужках. `{{battery}}` означає `msg.battery`, а не `msg.payload.battery`                                                                      |
| Switch-нода надсилає `msg` у всі виходи одночасно          | Перевірте налаштування — має бути `stopping after first match`, а не `checking all rules`                                                                              |
| Помилка `TypeError: Cannot read property of undefined`     | У Function-ноді звернення до `msg.payload.field`, коли `payload` ще не об'єкт. Перевірте структуру `msg`                                                               |
| Node-RED не запускається з терміналу: `node-red not found` | Перевірте, що Node-RED встановлено глобально (`npm install -g node-red`). На Linux/macOS може знадобитись `sudo`, на Windows — Command Prompt від імені адміністратора |
| `EADDRINUSE: address already in use :::1880`               | Порт 1880 зайнятий іншим процесом. Запустіть на іншому порту: `node-red --port 1881`                                                                                   |
| Знак emoji у Template виводиться як квадратик              | Переконайтесь, що файл потоку збережено у кодуванні UTF-8                                                                                                              |
| При експорті JSON-файл містить лише одну ноду              | Перед експортом виділіть усі ноди потоку: `Ctrl+A` або обведіть рамкою                                                                                                 |
