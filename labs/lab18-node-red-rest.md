**Мета:** Закріпити практичні навички побудови REST API засобами платформи Node-RED: створення HTTP-ендпоінтів за допомогою нод HTTP In та HTTP Response, обробка path- та query-параметрів запиту, формування JSON-відповідей з коректними кодами статусу, реалізація базової валідації вхідних даних.

## **Теоретичні відомості**

### **REST API та ресурси**

**REST API** — це стандартний спосіб обміну даними між клієнтом (браузер, HMI, контролер) і сервером через HTTP-запити над **ресурсами**. Кожен ресурс має унікальну URL-адресу, а HTTP-метод визначає дію над ним: `GET` — читати, `POST` — створювати, `PUT` — оновлювати, `DELETE` — видаляти. Дані передаються у форматі JSON. URL описує ресурс іменником у множині (`/api/actuators`), а не дію.

### **Ноди HTTP In та HTTP Response**

Нода **HTTP In** створює точку входу (endpoint) — вона стартує потік щоразу, коли надходить запит на заданий URL із заданим методом. Нода **HTTP Response** завершує обробку запиту й повертає результат клієнту. Без HTTP Response запит клієнта "зависне" до спрацювання timeout.

| Нода          | Основні налаштування     | Призначення                      |
| ------------- | ------------------------ | -------------------------------- |
| HTTP In       | `Method`, `URL`          | Прийняти запит, стартувати потік |
| HTTP Response | `Status code`, `Headers` | Повернути відповідь клієнту      |

Між ними зазвичай ставлять **Function**-ноду, що формує `msg.payload` (тіло відповіді) та `msg.statusCode` (код статусу). Якщо `msg.payload` — це JavaScript-об'єкт, Node-RED автоматично серіалізує його в JSON та встановить заголовок `Content-Type: application/json`.

### **Path- та query-параметри**

Клієнт передає дані в URL двома способами:

- **Path-параметр** — динамічна частина шляху. У налаштуваннях HTTP In позначається двокрапкою: URL `/api/actuators/:id`. Доступ: `msg.req.params.id`.
- **Query-параметр** — пара `ключ=значення` після знака `?`, використовується для фільтрації та сортування: `/api/actuators?type=pump`. Доступ: `msg.req.query.type`.

> **Важливо:** query-параметри **завжди приходять рядками**. Навіть `?limit=10` дає `msg.req.query.limit === "10"`. Для числа використовуйте `Number(...)`, для логічного значення порівнюйте з рядком `"true"`.

### **Тіло POST-запиту та JSON**

Для методів `POST` і `PUT` дані передаються в тілі запиту. Якщо клієнт вказав заголовок `Content-Type: application/json`, нода HTTP In **автоматично** розпарсить тіло й покладе результат у `msg.payload` як JavaScript-об'єкт. Без цього заголовка `msg.payload` залишиться рядком.

### **HTTP-коди статусу**

| Код     | Назва       | Коли повертати                                       |
| ------- | ----------- | ---------------------------------------------------- |
| **200** | OK          | Запит успішно виконано (стандартна відповідь на GET) |
| **201** | Created     | Створено новий ресурс (відповідь на POST)            |
| **400** | Bad Request | Запит некоректний (невалідні дані)                   |
| **404** | Not Found   | Ресурс за вказаним URL не існує                      |

Правило вибору: якщо клієнт надіслав щось неправильне → код **4xx**; якщо сервер не зміг обробити коректний запит → код **5xx**.

### **Базова валідація вхідних даних**

Будь-які дані від клієнта потенційно небезпечні: клієнт може надіслати рядок замість числа, `null`, об'єкт замість очікуваного поля або взагалі нічого. Сервер зобов'язаний самостійно перевірити наявність обов'язкових полів, їх типи, діапазони значень та формат. У разі помилки — повернути код `400` та зрозумілий опис проблеми, не виконуючи подальшу обробку.

## **Хід роботи**

У цій роботі ви побудуєте REST API для **реєстру виконавчих механізмів автоматизованої системи** (клапани, насоси, двигуни). API підтримуватиме читання списку пристроїв, отримання окремого пристрою за ідентифікатором, фільтрацію та додавання нового пристрою з валідацією.

Модель одного пристрою:

```json
{
  "id": "VLV-101",
  "name": "Запірний клапан лінії A",
  "type": "valve",
  "active": true
}
```

Поле `type` приймає одне з трьох значень: `valve` (клапан), `pump` (насос), `motor` (двигун).

### **Етап 1: Підготовка**

1.1. Запустіть Node-RED та відкрийте редактор у браузері за адресою `http://localhost:1880`.

1.2. Створіть нову вкладку потоку (натисніть `+` праворуч від назв вкладок) і назвіть її, наприклад, `REST API`.

1.3. **Створіть потік ініціалізації даних.** Реєстр пристроїв зберігатиметься у **flow-контексті**, щоб бути доступним усім ендпоінтам. Зберіть такий потік:

- нода **Inject** → нода **Function** (назвіть її `Ініціалізація реєстру`)

У ноді **Inject** увімкніть опцію **"Inject once after 0.1 seconds"** (поле `Inject once after ... seconds`) — це автоматично заповнить реєстр одразу після кожного Deploy.

Код Function-ноди `Ініціалізація реєстру`:

```javascript
flow.set("actuators", [
  {
    id: "VLV-101",
    name: "Запірний клапан лінії A",
    type: "valve",
    active: true,
  },
  { id: "PMP-201", name: "Насос охолодження", type: "pump", active: true },
  { id: "MTR-305", name: "Двигун конвеєра", type: "motor", active: false },
]);
return msg;
```

1.4. Натисніть **Deploy**. Потік ініціалізації спрацює автоматично та збереже три тестові пристрої у flow-контекст.

> **Примітка:** будь-які зміни в потоках застосовуються лише після натискання кнопки **Deploy** у правому верхньому куті редактора. Це найпоширеніша причина того, що "нічого не працює".

#### **Завдання 2: Ендпоінт окремого пристрою за ідентифікатором (path-параметр, 404)**

Створіть ендпоінт, що повертає **один** пристрій за його `id`, переданим у шляху URL. Якщо пристрій не знайдено — поверніть код `404`.

**Зберіть потік:**

- нода **HTTP In** → нода **Function** (`Пристрій за id`) → нода **HTTP Response**

**Налаштування HTTP In:**

| Параметр | Значення             |
| -------- | -------------------- |
| Method   | `GET`                |
| URL      | `/api/actuators/:id` |

**Код Function-ноди `Пристрій за id`** — допишіть логіку пошуку у позначеній зоні:

```javascript
const devices = flow.get("actuators") ?? [];
const id = msg.req.params.id;

// === ПОЧАТОК ВАШОГО КОДУ ===
// Знайдіть у масиві devices пристрій, у якого поле id дорівнює змінній id.
// Підказка: метод масиву find() з порівнянням item.id === id.
const device = ...;
// === КІНЕЦЬ ВАШОГО КОДУ ===

if (!device) {
  msg.statusCode = 404;
  msg.payload = { status: "error", message: `Пристрій '${id}' не знайдено` };
  return msg;
}

msg.statusCode = 200;
msg.payload = device;
return msg;
```

**Перевірка:** після **Deploy** відкрийте у браузері:

- `http://localhost:1880/api/actuators/PMP-201` — має повернути дані насоса (код 200);
- `http://localhost:1880/api/actuators/XXX-999` — має повернути повідомлення про помилку (код 404).

> Код статусу зручно перевірити у DevTools браузера: вкладка **Network**, оберіть запит і подивіться значення **Status**.

#### **Завдання 3: Фільтрація через query-параметри**

Доопрацюйте ендпоінт із **Завдання 1** так, щоб він підтримував фільтрацію за типом пристрою (`?type=`) та за станом активності (`?active=`).

Відкрийте Function-ноду `Список пристроїв` та замініть її код на наведений нижче, доповнивши логіку фільтрації:

```javascript
let devices = flow.get("actuators") ?? [];

// 1. Якщо передано type — залиште у devices лише пристрої цього типу.

  ...
}

// 2. Якщо передано active — порівняйте поле item.active з логічним значенням.
//    Зверніть увагу: query-параметр приходить рядком, тому activeParam === "true".
if (activeParam !== undefined) {
  const wantActive = activeParam === "true";
  ...
}
// === КІНЕЦЬ ВАШОГО КОДУ ===

  count: devices.length,
  data: devices
};

```

**Перевірка:** після **Deploy** відкрийте у браузері:

- `http://localhost:1880/api/actuators?type=pump` — лише насоси;
- `http://localhost:1880/api/actuators?active=false` — лише неактивні пристрої;
- `http://localhost:1880/api/actuators?type=valve&active=true` — активні клапани;
- `http://localhost:1880/api/actuators` (без параметрів) — повний список з усіх трьох пристроїв.

#### **Завдання 4: Ендпоінт створення пристрою (POST, валідація, 201/400)**

Створіть ендпоінт, що приймає новий пристрій у тілі POST-запиту, перевіряє коректність даних і додає пристрій до реєстру.

**Зберіть потік:**

- нода **HTTP In** → нода **Function** (`Додати пристрій`) → нода **HTTP Response**

**Налаштування HTTP In:**

| Параметр | Значення         |
| -------- | ---------------- |
| Method   | `POST`           |
| URL      | `/api/actuators` |

**Код Function-ноди `Додати пристрій`** — допишіть умови валідації у позначеній зоні:

```javascript
const data = msg.payload;
const errors = [];

// === ПОЧАТОК ВАШОГО КОДУ ===
// Додайте до масиву errors повідомлення, якщо:
//  - поле id відсутнє або не є непорожнім рядком;
//  - поле name відсутнє або не є непорожнім рядком;
//  - поле type не входить до списку allowedTypes.
const allowedTypes = ["valve", "pump", "motor"];

if (!data || typeof data.id !== "string" || data.id.trim() === "") {
  errors.push("Поле 'id' обов'язкове і має бути непорожнім рядком");
}

// ... аналогічно перевірте поле name ...

// ... перевірте, що data.type входить до allowedTypes (метод includes) ...
// === КІНЕЦЬ ВАШОГО КОДУ ===

// Якщо є помилки — повертаємо 400 Bad Request і припиняємо обробку
if (errors.length > 0) {
  msg.statusCode = 400;
  msg.payload = { status: "error", errors: errors };
  return msg;
}

// Дані валідні — додаємо пристрій до реєстру
const devices = flow.get("actuators") ?? [];
const newDevice = {
  id: data.id,
  name: data.name,
  type: data.type,
  active: data.active === undefined ? true : Boolean(data.active),
};

devices.push(newDevice);
flow.set("actuators", devices);

msg.statusCode = 201;
msg.payload = { status: "created", device: newDevice };
return msg;
```

**Перевірка.** POST-запит не можна надіслати з адресного рядка браузера, тому використайте один із способів.

**Спосіб А (рекомендований, у самому Node-RED).** Зберіть окремий тестовий потік:

- нода **Inject** → нода **Function** (`Тестовий запит`) → нода **HTTP Request** → нода **Debug**

Код Function-ноди `Тестовий запит`:

```javascript
msg.method = "POST";
msg.url = "http://localhost:1880/api/actuators";
msg.headers = { "Content-Type": "application/json" };
msg.payload = {
  id: "PMP-202",
  name: "Насос дренажу",
  type: "pump",
  active: true,
};
return msg;
```

У ноді **HTTP Request** встановіть: **Method** — оберіть зі списку варіант `- set by msg.method -`, **URL** — залишити порожнім (береться з `msg.url`), **Return** — `a parsed JSON object`. Відкрийте панель **Debug** (праворуч), натисніть кнопку зліва на Inject — у Debug з'явиться відповідь із кодом `201` та доданим пристроєм.

**Спосіб Б (через термінал, за допомогою `curl`):**

```bash
curl -X POST http://localhost:1880/api/actuators \
  -H "Content-Type: application/json" \
  -d "{\"id\":\"PMP-202\",\"name\":\"Насос дренажу\",\"type\":\"pump\"}"
```

> **Примітка для Windows:** запускайте команду в **Command Prompt (cmd)**, а не в PowerShell, де `curl` є псевдонімом іншої команди. Якщо термінал не розпізнає `curl`, скористайтесь Способом А.

**Обов'язково перевірте обидва сценарії:**

1. **Коректні дані** — відповідь має містити код `201` та об'єкт створеного пристрою.
2. **Некоректні дані** — надішліть запит без поля `id` або з `type` поза списком (наприклад, `"type":"sensor"`). Відповідь має містити код `400` та масив `errors`.

Після успішного POST відкрийте `http://localhost:1880/api/actuators` у браузері — у списку має з'явитися новий пристрій (поле `count` збільшиться).

## **Типові помилки та їх вирішення**

| Проблема                                             | Рішення                                                                                           |
| ---------------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| Запит у браузері "зависає" і не повертає відповіді   | Забута нода **HTTP Response** у кінці потоку — додайте її та з'єднайте з Function                 |
| `404 Not Found` для коректного URL                   | Перевірте написання URL у налаштуваннях HTTP In і натисніть **Deploy**                            |
| Список пристроїв порожній (`count: 0`)               | Не спрацював потік ініціалізації — перевірте опцію _Inject once_ та натисніть **Deploy**          |
| `msg.req.params.id` має значення `undefined`         | У налаштуваннях HTTP In пропущено `:id` у шляху — URL має бути `/api/actuators/:id`               |
| Фільтр `?active=true` нічого не повертає             | Query-параметр — рядок; порівнюйте `activeParam === "true"`, а не `=== true`                      |
| При POST `msg.payload` порожній або є рядком         | Клієнт не вказав заголовок `Content-Type: application/json` — додайте його до запиту              |
| Валідація пропускає некоректні дані                  | Перевіряйте і **наявність** поля, і його **тип** (`typeof`), не лише `if (data.id)`               |
| `Internal Server Error 500`                          | Виняток у Function-ноді — відкрийте лог Node-RED у терміналі та перевірте синтаксис коду          |
| Доданий через POST пристрій зникає після перезапуску | flow-контекст зберігається в пам'яті та скидається при рестарті — це очікувана поведінка в роботі |
