# **Лабораторна робота №19**

**Тема:** Дашборд моніторингу автоматизованої системи з MQTT-інтеграцією

**Мета:** Закріпити практичні навички побудови людино-машинного інтерфейсу (HMI) у Node-RED із застосуванням протоколу MQTT: підключення до брокера, обмін даними за моделлю publish/subscribe та відображення телеметрії на дашборді у реальному часі.

**Обладнання:** IBM-сумісні ПК

**Програмне забезпечення:** ОС Windows 10/11, macOS або Linux; Node.js (версія 18 LTS або новіша); Node-RED (версія 4.x); веб-браузер (Chrome, Firefox, Edge); палітра `@flowfuse/node-red-dashboard` (Dashboard 2.0); підключення до мережі Інтернет для доступу до публічного MQTT-брокера.

## **Теоретичні відомості**

### **Архітектура завдання**

У цій роботі один екземпляр Node-RED виконує дві ролі. Перший потік (**симулятор станції**) імітує польовий пристрій: він періодично формує показники та **публікує** їх на брокер. Другий потік (**дашборд**) **підписується** на ці показники й відображає їх оператору. Обидва потоки спілкуються не напряму, а через зовнішній брокер, що відтворює реальну схему збирання телеметрії в автоматизованих системах.

```
[Симулятор]  --publish-->  [ MQTT-брокер ]  --subscribe-->  [Дашборд]
```

### **Публічний брокер та унікальність тем**

Для роботи використовується публічний тестовий брокер, доступний усім без автентифікації:

| Параметр       | Значення                                              |
| -------------- | ----------------------------------------------------- |
| `Server`       | `broker.hivemq.com` (резервний: `test.mosquitto.org`) |
| `Port`         | `1883`                                                |
| Автентифікація | Не використовується (відкритий навчальний брокер)     |

> **Критично важливо:** публічний брокер одночасно використовують тисячі клієнтів з усього світу. Якщо двоє людей публікують у тему з однаковою назвою, їхні повідомлення змішуються. Тому **кожен студент зобов'язаний** включити у префікс своїх тем власний унікальний ідентифікатор (група та прізвище латиницею) і задати унікальний `Client ID`.

Структура теми у цій роботі (за схемою `<префікс>/<пристрій>/<параметр>`):

```text
akitr/<група>-<прізвище>/pumpstation/telemetry   ← телеметрія (публікує симулятор)
```

Приклад готового префікса: `akitr/kt31-shevchenko`. Далі у тексті він позначається як `<PREFIX>`.

### **Ноди MQTT та формат даних**

Нода `MQTT in` підписується на тему й формує `msg` із полями `topic`, `payload`, `qos`, `retain` щоразу, коли надходить повідомлення. Параметр **Output** визначає тип `payload`:

- **a String** — `payload` залишається рядком (для простих текстових повідомлень).
- **a parsed JSON object** — нода автоматично перетворює текст на об'єкт JavaScript. Тоді одразу доступні поля, наприклад `msg.payload.level`, без ручного `JSON.parse()`.

Нода `MQTT out` публікує вміст `msg.payload` у задану тему. Якщо `payload` є об'єктом, нода **автоматично** серіалізує його в рядок JSON — явний виклик `JSON.stringify()` не потрібен.

### **Віджети Dashboard 2.0**

| Віджет     | Призначення                                                  |
| ---------- | ------------------------------------------------------------ |
| `ui-gauge` | Стрілковий індикатор поточного значення в заданому діапазоні |
| `ui-chart` | Графік зміни числового значення у часі                       |
| `ui-text`  | Текстове відображення значення або стану                     |

Усі віджети розміщуються в ієрархії компоновки: `ui-base` → `ui-page` (сторінка) → `ui-group` (група) → віджет. Дашборд відкривається у браузері за адресою `http://localhost:1880/dashboard`.

## **Хід роботи**

У цій роботі ви побудуєте систему моніторингу **насосної станції водопостачання**. Симулятор формуватиме показники рівня води в резервуарі, тиску в магістралі та витрати, а дашборд відображатиме їх у реальному часі.

### **Етап 1: Підготовка**

1.1. Запустіть Node-RED та відкрийте редактор у браузері (`http://localhost:1880`).

1.2. Переконайтесь, що палітра Dashboard 2.0 встановлена. Відкрийте меню (☰) → **Manage palette** → вкладка **Install** і за потреби встановіть пакет `@flowfuse/node-red-dashboard`. Ноди `MQTT in` та `MQTT out` входять до ядра Node-RED і додаткового встановлення не потребують.

1.3. Визначте власний префікс тем за схемою `akitr/<група>-<прізвище>` (латиницею, без пробілів). Запишіть його — він знадобиться у MQTT-нодах.

### **Етап 2: Виконання завдань**

---

#### **⚙️ Завдання 1: Симулятор насосної станції (публікація)**

Побудуйте потік, що періодично формує телеметрію та публікує її на брокер.

**Структура потоку:**

```text
[Inject] --> [Function: генератор телеметрії] --> [MQTT out: .../telemetry]
```

1. Нода `Inject`: увімкніть **Repeat → interval**, інтервал **2 секунди**. Це джерело тактів симуляції.

2. Нода `Function` з назвою **«Генератор телеметрії»**. Доповніть наведений каркас (заповніть блок `TODO`):

```javascript
// TODO: сформувати об'єкт телеметрії у msg.payload з трьома полями:
//   level    - рівень води, ціле число у межах 60..80 (%),
//   pressure - тиск, число у межах 3.5..4.5 (атм), округлене до 2 знаків,
//   flow     - витрата, ціле число у межах 18..25 (м3/год).
// Підказки:
//   Math.random()           повертає число від 0 до 1;
//   Math.round(x)           округлює до цілого;
//   Number(x.toFixed(2))    округлює до 2 знаків після коми.
//
// msg.payload = { level: ..., pressure: ..., flow: ... };

// Нода MQTT out автоматично перетворить об'єкт на рядок JSON
return msg;
```

3. Нода `MQTT out`. Відкрийте її, біля поля **Server** натисніть олівець і створіть **конфігураційний вузол брокера**. На вкладці **Connection**:
   - **Server:** `broker.hivemq.com`, **Port:** `1883`
   - **Client ID:** задайте **власне унікальне** значення з вашим прізвищем, наприклад `nr-kt31-shevchenko-01` (не копіюйте приклад дослівно)

   Вкладку **Security** залиште порожньою (брокер відкритий).

   У самій ноді `MQTT out` задайте **Topic:** `<PREFIX>/pumpstation/telemetry`, **QoS:** `0`, **Retain:** вимкнено.

4. З'єднайте ноди. Тимчасово додайте `Debug` після `Function`. Натисніть **Deploy**. Під нодою `MQTT out` має з'явитися статус **connected**, а в панелі **Debug** кожні 2 секунди — об'єкт із трьома полями. Переконавшись у цьому, `Debug` можна вимкнути.

> **Примітка:** статус **connecting** замість **connected** найчастіше означає недоступність мережі або помилку в адресі брокера. Перевірте підключення до Інтернету та правильність написання `broker.hivemq.com`.

---

#### **📊 Завдання 2: Дашборд моніторингу (підписка та відображення)**

Побудуйте HMI, що приймає телеметрію та відображає її у реальному часі.

1. Нода `MQTT in`: оберіть створений раніше брокер, **Topic:** `<PREFIX>/pumpstation/telemetry`, **Output:** `a parsed JSON object`. Завдяки цьому режиму у наступних нодах буде доступний об'єкт із полями `msg.payload.level`, `msg.payload.pressure`, `msg.payload.flow`.

2. Від виходу `MQTT in` проведіть **три паралельні гілки**. У кожній гілці поставте ноду `Change`, що виділяє одне поле, і направте її до відповідного віджета:

| Гілка   | Нода `Change` (Set `msg.payload` to)       | Віджет дашборду                               |
| ------- | ------------------------------------------ | --------------------------------------------- |
| Рівень  | `msg.payload.level` (тип значення: `msg.`) | `ui-gauge` «Рівень води, %» (діапазон 0..100) |
| Тиск    | `msg.payload.pressure`                     | `ui-gauge` «Тиск, атм» (діапазон 0..5)        |
| Витрата | `msg.payload.flow`                         | `ui-chart` «Витрата, м³/год» (лінійний)       |

> **Як налаштувати `Change`:** правило **Set** → `msg.payload`, а у полі значення оберіть тип `msg.` та впишіть `payload.level` (для гілки рівня) тощо. Так із загального об'єкта виділяється одне значення для конкретного віджета.

3. У першому віджеті створіть ієрархію компоновки: новий `ui-base`, сторінку `ui-page` з назвою **«Насосна станція»** та групу `ui-group` **«Моніторинг»**. Решту віджетів розмістіть у цій самій групі.

4. Натисніть **Deploy** та відкрийте `http://localhost:1880/dashboard`. Ви маєте бачити, як кожні 2 секунди оновлюються індикатори рівня й тиску, а графік витрати будує лінію в часі.

### **Етап 3: Перевірка**

3.1. Переконайтесь, що ноди `MQTT out` та `MQTT in` мають статус **connected**.

3.2. Відкрийте дашборд (`http://localhost:1880/dashboard`) і впевніться, що індикатори рівня й тиску оновлюються в реальному часі, а графік витрати будується в часі.

3.3. На кілька секунд зупиніть симулятор (у ноді `Inject` змініть **Repeat** на `none` і зробіть **Deploy**), потім поверніть інтервал **2 секунди** та знову **Deploy** — простежте, як оновлення на дашборді зупиняється й відновлюється.

**Результат:** працюючий дашборд моніторингу насосної станції, що отримує телеметрію за протоколом MQTT та відображає її у реальному часі.

## **Типові помилки та їх вирішення**

| Проблема                                                   | Рішення                                                                                                              |
| ---------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| Нода MQTT показує статус **connecting** і не підключається | Перевірте підключення до Інтернету та точність адреси брокера (`broker.hivemq.com`, порт `1883`)                     |
| На дашборді з'являються «чужі» дані (не ваші значення)     | Ви використали неунікальну тему. Додайте до префікса свою групу й прізвище латиницею                                 |
| З'єднання раз у раз обривається (`connected` → `offline`)  | Двоє клієнтів підключилися з однаковим `Client ID` і відключають один одного. Задайте власний унікальний `Client ID` |
| `msg.payload.level` дорівнює `undefined`, віджети порожні  | У ноді `MQTT in` не обрано **Output: a parsed JSON object** — payload залишається рядком                             |
| Графік (`ui-chart`) не будується або показує помилку       | На вхід графіка має надходити **число**, а не об'єкт. Переконайтесь, що нода `Change` виділяє одне поле              |
| Віджети не з'являються на сторінці `/dashboard`            | Кожен віджет має бути прив'язаний до `ui-group`, а група — до `ui-page`. Перевірте ієрархію компоновки               |
| Симулятор не публікує дані                                 | У ноді `Inject` має бути увімкнено **Repeat → interval**; перевірте з'єднання нод та статус `MQTT out`               |
| Об'єкт телеметрії публікується як `[object Object]`        | Не викликайте `String(msg.payload)` перед `MQTT out` — нода серіалізує об'єкт у JSON автоматично                     |
